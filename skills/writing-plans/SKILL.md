# writing-plans — 대형 티켓 실행 계획 작성

ref: github.com/obra/superpowers (writing-plans skill)

복잡한 티켓을 2-5분 단위 실행 태스크로 분해하고, 표준 계획 문서를 작성합니다.

## 트리거

- `/brainstorm` 스킬에서 설계 승인 후
- AC 4개 이상이거나 파일 변경이 6개 이상 예상되는 티켓
- 아키텍처 결정이 포함된 작업

## 절대 규칙

> **계획 없이 복잡 티켓 시작 금지**
> 계획 문서 작성 → TASK.md 티켓 등록 → 구현 시작 순서 준수.

## 실행 순서

### Step 1: 컨텍스트 확인

1. `/brainstorm` 승인 결과 또는 TASK.md 해당 티켓 읽기
2. 관련 기존 코드/파일 탐색 (영향 범위 파악)
3. 공유 패키지 목록 확인 — 재사용 가능한 것이 있으면 명시
4. `tasks/lessons.md` — 유사 작업 실수 확인

### Step 2: 계획 문서 작성

파일: `docs/plans/YYYY-MM-DD-<feature-name>.md`

```markdown
# [기능명] 구현 계획

날짜: YYYY-MM-DD
관련 티켓: T[번호]
예상 소요: [N]개 태스크

## 목표

[한 문장으로 무엇을 달성하는가]

## 아키텍처 결정

[설계 방식 및 이유 — /brainstorm 결과 반영]

## 영향 범위

- 변경 파일: [파일 목록]
- 공유 패키지: [활용 패키지]
- 사이드이펙트 위험: [있으면 명시]

## 실행 태스크 (2-5분 단위)

각 태스크는 다음 사이클을 따릅니다:
TEST FAIL → IMPLEMENT → TEST PASS → COMMIT

### Task 1: [이름]

- 작업: [구체적으로 무엇을]
- 파일: [변경 파일]
- 검증: [어떻게 확인하는가]
- 예상 소요: 2-5분

### Task 2: [이름]

...

## 완료 기준

- [ ] 모든 태스크 완료
- [ ] `pnpm validate` PASS
- [ ] TASK.md AC 전부 체크
```

### Step 3: TASK.md 등록

계획 문서를 기반으로 TASK.md에 티켓 등록 (이미 있으면 AC 보완).
계획 문서 경로를 티켓에 참조 링크로 추가.

### Step 4: 실행 시작

계획 문서의 Task 1부터 순서대로 실행.
각 태스크 완료 시 계획 문서에 ✅ 표시.

## 계획 변경 시

- 계획 이탈 발견 → 즉시 멈추고 계획 문서 업데이트
- 추가 태스크 필요 → 계획 문서 끝에 추가 후 사용자에게 알림
- 추측으로 계속 진행 금지
