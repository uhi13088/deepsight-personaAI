# 품질 체크리스트

검증 실행 후, 아래 항목을 추가로 확인합니다.

## 코드 품질

- [ ] TypeScript strict 모드 위반 없음
- [ ] `any` 타입 사용 없음
- [ ] API 응답 형식 준수: `{ success, data?, error? }`
- [ ] 커밋 메시지 형식 준수: `type(scope): message`

## 변경 영향 범위

- [ ] 변경한 파일만 수정됨 (불필요한 파일 변경 없음)
- [ ] 기존 테스트가 깨지지 않음
- [ ] import 경로가 올바름

## lessons.md 교차 확인

- [ ] `tasks/lessons.md`의 각 카테고리 확인
- [ ] 현재 변경 사항이 기록된 교훈을 위반하지 않음
- [ ] 새로 배운 교훈이 있다면 기록했는가?

## Staff Engineer 자문

> "이 코드가 PR로 올라왔을 때, Staff Engineer가 승인할 수 있는가?"

- [ ] 과도한 추상화 없음
- [ ] 임시 fix 아닌 근본 해결
- [ ] 테스트가 의미 있는 케이스를 커버

## 2단계 코드 리뷰 (ref: obra/superpowers subagent-driven-development)

복잡 티켓(AC 3개 이상 또는 파일 5개 이상 변경) 완료 시 2단계 검토를 수행합니다.

### Stage 1: 사양 검증 (Spec Verification)

> "구현한 것이 요청한 것과 일치하는가?"

- [ ] TASK.md 해당 티켓의 모든 AC를 구현했는가?
- [ ] AC 요구사항을 충족했음을 테스트 출력으로 증명했는가?
- [ ] 요청하지 않은 기능을 추가하지 않았는가? (over-engineering 금지)
- [ ] API 변경이 있다면 `docs/api/` 문서도 업데이트했는가?
- [ ] DB 스키마 변경이 있다면 마이그레이션 파일이 있는가?

### Stage 2: 코드 품질 검증 (Code Quality Review)

> "구현 방식이 올바른가?"

- [ ] 기존 공유 패키지(`@deepsight/*`)를 활용했는가? (재구현 금지)
- [ ] 타입 안전성 — `any`, `as` 캐스팅 남용 없음
- [ ] 에러 처리 — API 경계에서 적절히 처리, 내부 코드는 신뢰
- [ ] 테스트 케이스가 엣지케이스를 커버하는가?
- [ ] 사이드이펙트 — 변경이 다른 모듈에 예상치 못한 영향을 주는가?
- [ ] 성능 — 불필요한 DB 쿼리, 무한 루프 가능성 없는가?
