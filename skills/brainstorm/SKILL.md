# brainstorm — 설계 승인 전 구현 금지

ref: github.com/obra/superpowers (brainstorming skill)

새로운 기능/변경 작업 시작 전에 설계를 승인받고 진행합니다.

## 트리거

- 사용자가 새로운 기능, 아키텍처 변경, 3단계 이상 작업을 요청할 때
- TASK.md에 새 티켓을 추가하기 전
- Plan Mode 진입 전

## 절대 규칙

> **설계 승인 없이 코드 한 줄도 작성 금지**
> 간단한 작업도 간단한 설계 먼저. 예외 없음.

## 실행 순서

### Step 1: 프로젝트 컨텍스트 파악

1. `TASK.md` 현재 상태 확인 — 관련 티켓/배경 있는지 파악
2. `CLAUDE.md` 금지 사항 및 패키지 목록 재확인
3. 관련 기존 코드/파일 탐색 — 이미 비슷한 게 있으면 재구현 금지
4. `tasks/lessons.md` — 유사 작업에서 과거 실수 확인

### Step 2: 명확화 질문 (1개씩, 순서대로)

목표가 불명확하면 질문합니다. **한 번에 1개 질문만**. 답변 받고 다음 질문.

명확화할 항목:

- 이 기능이 해결하려는 근본 문제는?
- 성공 기준(AC)은 어떻게 정의할 것인가?
- 범위 — 어디까지가 이번 작업이고 어디서부터 다음 티켓인가?
- 의존성 — 다른 팀/앱/서비스에 영향을 주는가?

### Step 3: 방식 제안 (2~3가지)

각 방식에 대해:

```
방식 A: [이름]
- 접근: [어떻게]
- 장점: [무엇이 좋은가]
- 단점/리스크: [무엇이 문제인가]
- 예상 티켓 수: [N개]
```

### Step 4: 설계 승인 요청

추천 방식을 명시하고 승인 요청.

```
추천: 방식 [X]
이유: [한 줄 이유]

승인하시면 `/writing-plans`로 상세 계획을 작성합니다.
```

### Step 5: 설계 문서화 (승인 후)

승인된 설계를 TASK.md에 Phase/티켓으로 등록.
→ `/writing-plans` 스킬 호출

## 막힘 기준

- 요구사항이 여전히 불명확하면 → 질문 반복 (구현 시작 금지)
- 기존 코드와 충돌 가능성 → 탐색 후 보고 (추측으로 진행 금지)
