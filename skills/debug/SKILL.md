# debug - 구조적 디버깅

에러나 버그를 체계적으로 진단하고 해결합니다.

## 트리거

- 사용자가 "버그", "에러", "안돼", "debug" 등을 보고할 때
- 테스트 FAIL 시 원인 분석이 필요할 때

## 실행 순서

### Step 1: 증상 수집

1. 에러 메시지 / 스택 트레이스 확인
2. 재현 조건 파악 (어떤 상황에서 발생하는가?)
3. 최근 변경 사항 확인 (`git diff`, `git log -5`)

### Step 2: 가설 수립

- 증상을 기반으로 원인 가설 3개 이내로 수립
- 각 가설에 대해 검증 방법 명시
- 예:
  ```
  가설 1: DB 연결 타임아웃 → 검증: 로그에서 connection timeout 확인
  가설 2: 환경변수 누락 → 검증: .env.example과 비교
  가설 3: 최근 커밋 사이드이펙트 → 검증: git bisect 또는 revert 테스트
  ```

### Step 3: 가설 검증 (좁혀가기)

- 가설별로 순차 검증
- 로그, 테스트, 코드 리딩으로 원인 좁히기
- **밀어붙이지 않기**: 2번 이상 같은 접근이 실패하면 → 멈추고 재계획

### Step 4: 근본 원인 해결

- 임시 fix 금지 (증상만 숨기는 수정 불가)
- 근본 원인을 해결하는 코드 작성
- 해당 버그를 방지하는 테스트 추가

### Step 5: 검증 및 기록

- 수정 후 전체 테스트 실행 (`pnpm test`)
- **lessons.md에 교훈 기록** (같은 버그 재발 방지)
  - 형식: `- [날짜] 카테고리: 교훈 (관련 티켓: T#)`

## 출력

```
## 디버깅 결과

### 증상
- (에러 메시지 요약)

### 원인
- (근본 원인)

### 수정
- (변경 사항 요약)

### 방지책
- (추가한 테스트 또는 lessons.md 기록)
```

## 성공 기준

- 근본 원인 파악 및 해결 (임시 fix 아님)
- 해당 버그 방지 테스트 추가
- 테스트 전체 PASS
- lessons.md에 교훈 기록
